# 语言变化迁移样本标注规范 v1

## 1. 目标

这份规范服务于 Porta-Bench 场景二：`语言变化带来的迁移`。

当前覆盖 5 类 subtype：
- `py2_py3`
- `cpp_python`
- `java_python`
- `python_cpp`
- `python_java`

## 2. taxonomy

| subtype | migration_type | 含义 |
| --- | --- | --- |
| `py2_py3` | `version_migration` | 同一种语言的版本迁移 |
| `cpp_python` | `cross_language_migration` | C/C++ 到 Python |
| `java_python` | `cross_language_migration` | Java 到 Python |
| `python_cpp` | `cross_language_migration` | Python 到 C/C++ |
| `python_java` | `cross_language_migration` | Python 到 Java |

## 3. implementation_scope

这个字段用来区分迁移范围：
- `partial_feature_migration`
- `full_repo_translation`
- `not_applicable`

### `partial_feature_migration`

适用于：
- `r0` 中，目标功能主要由源语言实现
- `rn` 中，这个功能被目标语言实现替代、承接或主导
- 仓库整体仍然是混合形态

### `full_repo_translation`

适用于：
- `r0` 中，仓库主体或核心产品几乎完全是源语言实现
- `rn` 中，仓库主体或核心产品整体切换成目标语言实现

### `not_applicable`

适用于：
- `py2_py3` 这类同语言版本迁移

## 4. logic_equivalence_scope

这个字段用来判断“是不是同一逻辑的翻译或替代”。

可选值：
- `same_logic_translation`
- `partial_logic_replacement`
- `unclear_logic_mapping`

### `same_logic_translation`

适用于：
- 修改前后是同一个功能、同一个算法或同一个逻辑链条
- 只是实现语言发生变化

### `partial_logic_replacement`

适用于：
- 只迁移或替换了其中一个功能片段、子模块、hot path 或 runtime 组件

### `unclear_logic_mapping`

适用于：
- 看不清是否为同一逻辑翻译
- 改动太发散，不适合直接作为高质量正例

## 5. 必要纳入条件

1. PR 的根因必须直接来自语言或语言版本变化。
2. 必须能标出 source/target language 或 version。
3. 改动主体应触及源码、运行时语义、构建适配或测试适配。
4. 需要保留原始数据：
   - 仓库创建时间
   - PR 提出时间
   - stars 数
   - changed files 摘要
5. 样本最好具备 issue 改写空间和后续测试验证空间。

## 6. 全局排除条件

1. 纯文档 PR
2. 纯 CI PR
3. 纯 dependency bump PR
4. 机器人自动生成 PR
5. 与语言迁移无关的普通 bugfix / refactor

## 7. 各 subtype 判定重点

### `py2_py3`

正例：
- 明确是 Python 2 到 Python 3 迁移
- 改动涉及典型迁移点，如 `xrange`、`iteritems`、`raw_input`、`basestring`、`2to3`

负例：
- 只是支持 Python 3.11/3.12/3.13/3.14
- 只是 CI、依赖、文档

推荐填写：
- `implementation_scope = not_applicable`
- `logic_equivalence_scope = same_logic_translation`

### `cpp_python` / `java_python` / `python_cpp` / `python_java`

正例：
- PR 明确把已有实现迁移、替换或重写为另一种语言实现
- 修改前后仍然是同一个功能模块或工具
- 文件层面能看到源语言文件与目标语言文件的迁移关联

负例：
- 只是 binding / wrapper / SDK / bridge / gateway
- 只是改构建系统
- 只是新增接口，但没有对应逻辑替换
- 大规模重写到已经不适合作为单个 benchmark task

## 8. CSV 字段怎么填

| 字段 | 填写建议 |
| --- | --- |
| `manual_label` | `positive` / `negative` / `uncertain` |
| `migration_type` | `version_migration` / `cross_language_migration` |
| `implementation_scope` | `partial_feature_migration` / `full_repo_translation` / `not_applicable` |
| `logic_equivalence_scope` | `same_logic_translation` / `partial_logic_replacement` / `unclear_logic_mapping` |
| `source_language` / `target_language` | 如 `python`、`c++`、`java` |
| `source_version` / `target_version` | 如 `2`、`3`、`3.12` |
| `repo_created_at` | 从 metadata 中抄，不要自己估计 |
| `pr_created_at` | 从 metadata 中抄，不要自己估计 |
| `repo_stars` | 从 metadata 中抄 |
| `reproducible` | `yes` / `no` / `unknown` |
| `issue_rewrite_ready` | `yes` / `needs_check` / `no` |
| `leakage_risk` | `low` / `medium` / `high` |

## 9. 标注顺序

1. 先看标题和 PR body，判断根因是否真是语言迁移。
2. 再看 changed files，确认是否真的触及代码和测试。
3. 再填 source/target、migration_type、implementation_scope、logic_equivalence_scope。
4. 最后评估是否可复现、是否适合改写成 issue、是否存在泄漏风险。

## 10. 一致性要求

1. 两人先对同一批 20 条样本做重叠标注。
2. 如果一致率低于 85%，先修订规范，再继续扩大样本。
3. `uncertain` 样本不直接丢弃，进入二次复核。
