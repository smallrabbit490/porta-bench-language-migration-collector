你可以把我现在做的东西，理解成一个“把 GitHub 上的 PR 变成 Porta-Bench 数据集”的小工厂。

**先说人话版**
PPT 里讲的是一条大流程：`GitHub -> Query & Collect -> Conditional Filter -> PR Filter -> Execution Filter -> Target Repo`。  
我现在真正落地完成的是这条流程的前半段，重点服务你们的“语言变化带来的迁移”任务，也就是：

- `py2 -> py3`
- `cpp -> python`

目前它已经能做的是：
- 去 GitHub 按 query 搜 PR
- 先做一轮自动过滤
- 把候选 PR 的元数据和代码快照保存下来
- 导出人工审核表给你们手工筛
- 把筛完的结果打包成统一的数据集文件

所以，你现在手里的不是“一个爬虫脚本”，而是一个“数据收集流水线”。

**技术实现是什么**
这里没有用 Django、Flask 这种“网站框架”。  
更准确地说，它是一个：

- `Python 命令行工具`
- `配置驱动`
- `ETL 数据流水线`
- `Human-in-the-loop（人工参与审核）`

底层技术主要是：
- `argparse`
  负责命令行参数
- `requests + urllib3 Retry`
  负责调 GitHub API 和重试
- `subprocess + git`
  负责拉代码、切 commit、保存 `r0/rn`
- `json/jsonl/csv`
  负责不同阶段的数据落盘
- `pathlib`
  负责目录和文件管理

脚本内部的结构也比较清楚：
- [GitHubClient](D:/thecourceofdasi/TeacherWangWork/SWEbench（zzc）/需求文档/download_repo.py#L219)
  专门负责和 GitHub API 通信
- [PortaBenchCollector](D:/thecourceofdasi/TeacherWangWork/SWEbench（zzc）/需求文档/download_repo.py#L347)
  专门负责把 5 个阶段串起来

Token 这块我也做了适配：

- 还支持你学长说的“直接传一个 query 字符串”，也就是 `--query`

**你可以怎么理解整个框架**
把它想成这张图就行：

```text
GitHub PR 搜索
   ↓
collect
先把“像语言迁移的 PR”捞出来
   ↓
enrich
补全 PR 详情 + 下载前后代码状态 r0 / rn
   ↓
export-review
导出人工审核表
   ↓
你们手工判断：这条 PR 到底是不是目标迁移
   ↓
apply-review
把人工标签写回系统
   ↓
package
生成最终数据集和统计文件
```

其中：
- `r0` = 改动前的代码状态
- `rn` = 改动后的代码状态

这两个就是 PPT 里很关键的那对“前后仓库状态”。

**你现在最该学会的 3 个点**
- 爬虫不是“把东西爬下来”就结束了，真正重要的是“筛选逻辑”和“数据格式”
- query 很重要，但不能只靠 query，后面还要有自动过滤和人工审核
- 你们这类任务本质上是“数据工程 + 人工标注”，不是单纯写一个脚本
